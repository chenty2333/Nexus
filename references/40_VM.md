## Current shape

- `crates/axle-mm` already holds the bulk of VM metadata logic in a host-testable crate.
- The kernel wraps that metadata in `VmDomain`, page-table management, and bootstrap userspace mapping code.
- VM control-plane object entry points now live in `kernel/axle-kernel/src/object/vm.rs`.
- `object.rs` still defines VMO / VMAR object records, but create/read/write/map/unmap/protect no longer live in the root object module.
- Current VM work in-tree is already beyond pure roadmap prose:
  - VMO / VMAR metadata exists
  - lazy anonymous and lazy VMO fault paths exist
  - COW exists
  - channel page-loan exists in a limited full-page form
- TLB maintenance is now explicitly split into:
  - relaxed commits, which may stay epoch-lazy
  - strict commits, which synchronize active CPUs before return or before frame reuse
- Known strict invalidations now keep one merged per-address-space page-range plan before
  escalating to full-flush shootdown, and CPUs that re-enter an address space later replay that
  same pending plan before falling back to full local flush.
- `ShootdownBatch` now automatically promotes to a full-context flush (`FlushOp::All`) when the
  number of individual page invalidations exceeds a per-batch threshold, preventing O(n^2)
  deduplication cost in large unmap or protect operations.
- The current bootstrap-friendly CPU tracking for strict TLB sync is intentionally an internal VM
  detail.
  - the implementation still uses a fixed-size active-CPU tracker today
  - that limit is not meant to leak into syscall or object-layer contracts
- Frame reuse now follows a stronger rule than invalidation:
  - lazy invalidate is allowed for the relaxed class
  - lazy reuse is not allowed; bootstrap frame recycling waits for the relevant strict barrier
